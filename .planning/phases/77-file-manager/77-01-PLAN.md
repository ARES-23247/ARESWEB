<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → API | Untrusted file uploads and query params cross here |
| API → R2 | Validated files uploaded |
| API → D1 | File metadata written |
| API → Google Drive API | Service account token authenticates outbound requests |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-77-01 | Spoofing | File upload | mitigate | MIME type validation via allowlist (pdf, docx, xlsx, pptx, txt) per D-01; magic byte validation in handler |
| T-77-02 | Tampering | File size | mitigate | Size limit enforced (25MB default via MAX_FILE_SIZE_MB env var) per D-03; reject oversized files before processing |
| T-77-03 | Information Disclosure | Error messages | mitigate | Generic errors to client; detailed errors in server logs only; no R2 keys exposed |
| T-77-04 | Denial of Service | File uploads | mitigate | Size limit prevents memory exhaustion; sequential processing per Phase 76 pattern |
| T-77-05 | Spoofing | Drive import | mitigate | Fetch file details from Google Drive API, don't trust user-provided metadata; use authenticated service account |
| T-77-06 | Injection | Filename sanitization | mitigate | Sanitize filenames per D-05 (remove special chars, keep extension); use generated R2 keys not user paths |
</threat_model>
