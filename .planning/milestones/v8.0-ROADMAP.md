## Goal

Eliminate the `any` type annotation on the Hono RPC client (`src/api/honoClient.ts`) and achieve full end-to-end type inference from server route handlers through the `hc<AppType>()` client to frontend API calls.

Currently, every `client.tasks.$get()` call is **completely untyped** because:

1. `export const client: any = hc<AppType>(...)` â€” the client itself is typed `any`
2. The `AppType` exported from `[[route]].ts` only captures routes defined directly on `apiRouter`, not the 30+ sub-routers mounted via `.route()`

## Background & Root Cause Analysis

### Problem 1: Imperative `.route()` breaks type chain

Hono's type inference requires **chained** `.route()` calls so TypeScript can accumulate the composite type:

```typescript
// âťŚ Current (types lost):
const apiRouter = new OpenAPIHono<AppEnv>();
apiRouter.route("/tasks", tasksRouter);   // imperative â€” TS can't track
apiRouter.route("/posts", postsRouter);   // imperative â€” TS can't track
export type AppType = typeof apiRouter;   // only knows about directly-defined routes

// âś… Required (types preserved):
const apiRouter = new OpenAPIHono<AppEnv>()
  .route("/tasks", tasksRouter)
  .route("/posts", postsRouter);
export type AppType = typeof apiRouter;   // knows about all sub-routes
```

**Complication**: The current `[[route]].ts` interleaves middleware `.use()` calls between route mounts (CSRF, CORS, rate limiting on lines 88â€“178). All middleware must be applied before chaining routes in a single expression.

### Problem 2: Sub-router handlers cast responses to `any`

Every OpenAPIHono handler in the codebase ends with:

```typescript
return c.json({ tasks: formattedTasks } as any, 200);
```

This `as any` exists because:
- `OpenAPIHono.openapi()` enforces strict Zod response schema matching
- The actual response objects include computed/transformed fields (`assignees_json`, `creatorName`, `zulipStream`) that don't structurally match the declared Zod schema
- Without `as any`, TypeScript reports type errors on every handler return

Even if Problem 1 is fixed, these `as any` casts would propagate `any` through the RPC client anyway.

### Current Mitigation

Type safety is currently maintained through the **Zod wrapper layer** in `src/api/*.ts`:

```
Frontend Component â†’ src/api/tasks.ts (Zod-typed) â†’ client.tasks.$get() (untyped) â†’ Server
```

The `unwrapResponse<T>()` generic + Zod schemas on both sides provide the real type contracts. The `hc()` client is effectively just a path builder + fetch helper.

## Proposed Phases

### Phase 1: Restructure `[[route]].ts` for chained `.route()`

- Move all middleware (CORS, CSRF, rate limiting, usage metrics) into a pre-configured base router
- Chain all 30+ `.route()` calls in a single expression
- Export `AppType = typeof apiRouter` from the fully-chained instance
- Verify `hc<AppType>` now infers sub-router paths (test with a few routes)

### Phase 2: Align handler return types with Zod schemas

- Audit all `as any` casts in `c.json()` returns across `functions/api/routes/`
- For each handler, ensure the returned object structurally matches the declared Zod response schema
- This may require adjusting Zod schemas to be more permissive or adding transform steps
- Remove `as any` casts one router at a time

### Phase 3: Remove `any` from `honoClient.ts`

- Remove `: any` annotation from `export const client`
- Verify full type inference from client calls
- Update `src/api/*.ts` wrappers to leverage inferred types instead of manual generics
- Run `npx tsc --noEmit` to confirm zero regressions

### Phase 4: Performance validation

- Measure TypeScript compiler performance with the deep composite type
- If IDE responsiveness degrades significantly, consider splitting `AppType` by domain
- Document any TS compiler workarounds needed
